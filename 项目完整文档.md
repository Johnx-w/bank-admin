## 一、项目概况

### 1.1 项目定位

银行后台管理系统（Bank Admin System），面向银行内部运营人员的管理平台，处理用户管理、交易审核、账户操作、权限控制等核心金融业务。

这不是一次性 Demo 或原型，而是预计维护 3 年以上的企业级前端项目。

### 1.2 目标岗位

该项目的面试目标为长亮科技（深圳）银行前端/全栈岗位，薪资 12k-20k。
JD 关键要求：React、Electron、银行项目经验、TypeScript、全栈意识。

## 三、架构设计

### 3.1 分层架构

```
+-----------------------------+
|       Electron Shell        |  （可选，未来批次）
+-----------------------------+
|    Presentation Layer       |
|  Pages / Components / Layouts / Router |
+-----------------------------+
|    Application Layer         |
|  Zustand Stores / Custom Hooks |
+-----------------------------+
|        API Layer             |
|  Axios / Interceptors / api/* |
+-----------------------------+
|        Mock Layer            |
|  MSW Handlers / Mock Data   |
+-----------------------------+
|        Data Layer            |
|  TypeScript Types / Utils   |
+-----------------------------+
```

### 3.2 数据流

用户操作（点击/表单提交） -> 页面组件 -> API 函数（Axios + Token 拦截器）-> Mock 层（MSW）或真实后端 -> 状态更新（Zustand）-> UI 重渲染

两层守卫：AuthGuard（未登录 -> /login）+ PermissionGuard（无权限 -> 403）

## 五、关键设计决策

### 5.1 为什么选 Zustand 不选 Redux？

Zustand 样板代码少，没有 action type / reducer / dispatch 那一套，TypeScript 支持自然。
中后台系统的状态复杂度完全够用。

### 5.2 为什么用 MSW 做 Mock？

MSW 在 Service Worker 层拦截 HTTP 请求，开发时就像调真实后端。
切换到真实接口时只需改 baseURL，不需要动业务代码。

### 5.3 为什么用 Ant Design 不用 shadcn/ui？

Ant Design 是国内中后台/金融系统的事实标准，组件丰富（Table/Form/Modal 等 60+）。
shadcn/ui 偏展示型网站，与银行项目匹配度低。

### 5.4 为什么没有真实后端？

这是一个纯前端岗位，面试目标是展示前端能力。
MSW 模拟了全部接口，开箱即跑，不依赖后端服务。

### 5.5 Electron 方案

Electron 作为外壳，核心 Web 逻辑与 UI 层代码完全解耦。
后期改 B/S 架构只需移除 electron/ 目录。

## 六、编码规范

### 6.1 质量约束

1. TypeScript 严格模式：禁止 any，所有函数必须明确定义入参和出参类型
2. 命名规范：camelCase（变量/函数）、UPPER_SNAKE_CASE（常量）、PascalCase（组件）
3. 注释规范：所有导出函数必须有 JSDoc 注释；复杂业务逻辑必须有行内注释
4. 单一职责：每个函数只做一件事，超过 50 行必须拆分
5. 错误处理：所有异步操作必须有 try-catch，并展示友好错误提示
6. 可扩展性：枚举/常量集中定义在 constants.ts，禁止硬编码魔法数字

### 6.2 测试规范（Vitest）

1. 使用 vi.mock() 在模块级 mock（自动 hoist）
2. 模块级副作用在 beforeAll 中一次性 import，测试复用已注册的引用
3. 在 mock factory 外部声明变量捕获调用参数
4. beforeEach 只做数据清空，不清空 mock 闭包引用
5. 拦截器通过控制输入数据改变行为，不重置函数本身
6. 使用浏览器 API 的测试必须设置 environment: jsdom
7. 单文件 jsdom 在文件顶部加 // @vitest-environment jsdom
8. 优先用纯函数的输入/输出验证
9. 每个测试独立——清理自己的输入数据，断言自己的输出
10. 边界条件：空值、不存在、网络错误

### 6.3 React 最佳实践

遵循 Vercel React Best Practices 核心规则：

- rerender-memo：usePagination 的 onChange/reset 用 useCallback 记忆化
- rerender-derived-state-no-effect：skip 值在渲染时推导（(page-1)*pageSize）
- rendering-conditional-render：使用三元表达式而非 && 做条件渲染
- bundle-barrel-imports：直接路径导入，不使用 barrel 文件
- js-early-exit：StatCard loading 时直接返回 Spin，ConfirmModal 无 title 时直接 return

## 十二、常见问题

**Q：为什么运行后页面空白？**
A：检查浏览器控制台。常见原因：MSW 未正常注册、Ant Design 版本不兼容。

**Q：MSW 启动失败怎么办？**
A：确认 public/mockServiceWorker.js 存在。如果不存在，运行 npx msw init public/。

**Q：如何切换到真实后端？**
A：1) 注释 main.tsx 中的 startWorker()；2) 修改 constants.ts 中的 API_BASE_URL。

**Q：为什么 npm test 报错？**
A：确保所有依赖已安装（npm install）。jsdom 相关错误运行 npm install -D jsdom。

**Q：React Router v7 API 兼容吗？**
A：v7 的库模式完全向后兼容 v6 经典 API（BrowserRouter、Routes、useNavigate 等）。
