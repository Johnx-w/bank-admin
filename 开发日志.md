> **开发周期：**
### 5.9 Docker 容器化

| 文件 | 说明 |
|------|------|
| Dockerfile | 开发模式镜像：node:22-alpine + npm ci + vite --host 0.0.0.0，暴露 5173 |
| docker-compose.yml | 单服务编排：build . → port 5173:5173 → restart unless-stopped |
| .dockerignore | 排除 node_modules/.git/dist 避免无效层缓存 |

**关键认知：**
- Dockerfile 是"配方"，按层构建镜像，每层是文件系统快照
- 镜像 = 迷你 Linux + Node + 项目代码 + 依赖的压缩包
- 容器 = 镜像解压后运行的隔离进程
- 端口映射 (5173:5173) = 把容器内部端口桥接到宿主机
- 开发模式直接跑 Vite dev server，MSW Mock 正常工作，等价于本地 npm run dev

**对比生产模式（第一版失败）：**
- 第一版 multi-stage：build → nginx + dist 静态文件，不走 MSW，登录不可用
- 最终方案（开发模式）：node:22-alpine + vite --host 0.0.0.0，与本地开发体验一致 2026-07-08 ~ 2026-07-10（3天）
> **技术栈：** React 19 + TypeScript 6 + Vite 8 + Ant Design 6 + Zustand 5 + MSW 2 + Vitest 4
> **最终规模：** 87 个源文件，约 8,000 行代码，17 个测试文件（147 条用例），TypeScript 严格模式零 any，编译零错误

## 一、项目启动与脚手架搭建

### 1.1 技术选型决策

| 决策项 | 选择 | 原因 |
|--------|------|------|
| UI 框架 | React 19 | JD 要求 React，生态最成熟 |
| 构建工具 | Vite 8 | 比 CRA 快 10x，原生 ESM |
| 语言 | TypeScript 6 strict | 银行项目对类型安全要求极高 |
| 组件库 | Ant Design 6 | 中后台系统事实标准，ProTable 省大量工作量 |
| 状态管理 | Zustand 5 | 比 Redux 轻 90%，API 直觉化 |
| Mock 方案 | MSW 2 | Service Worker 层拦截，浏览器 DevTools 可见 |
| 测试框架 | Vitest 4 | 与 Vite 共用配置，零额外配置成本 |
| 图表 | ECharts 6（按需引入） | 金融报表图表需求，Tree-shaking 后打包体积仅 +80KB |
| HTTP 客户端 | Axios | 拦截器机制天然适合 Token 注入和统一错误处理 |

### 1.2 脚手架命令

```bash
npm create vite@latest bank-admin -- --template react-ts
```

创建时选择了 Oxlint（而非 ESLint）作为 linter，因为 Oxlint 速度比 ESLint 快 50-100x。

## 二、质量规范先行（instructions.md）

在写任何功能代码之前，先确立了项目的代码规范文档 `instructions.md`，内容要点：

1. **TypeScript 严格模式**：禁止 `any`，所有函数明确定义入参和出参类型
2. **命名规范**：camelCase / UPPER_SNAKE_CASE / PascalCase / kebab-case
3. **JSDoc 注释规范**：所有导出函数必须标注功能、参数、返回值
4. **单一职责**：函数不超过 50 行
5. **错误处理**：异步操作必须 try-catch，向用户展示友好提示
6. **常量管理**：魔法数字 / 枚举值集中在 `constants.ts`
7. **React 渲染优化规则**：useCallback 记忆化、条件渲染用三元而非 `&&`、early exit 模式

这个文档对后续所有批次开发起到了"宪法"级别的作用，确保换对话或加人时质量不滑坡。

### 第五批：通用组件（7 文件，~350 行）

**时间：** 2026-07-09 上午
**完成内容：**

| 组件 | 功能 |
|------|------|
| `ErrorBoundary` | React Error Boundary，捕获子树渲染错误，显示友好错误页面 |
| `EmptyState` | 空状态组件，支持自定义图标和描述文案 |

- `PermissionBtn` 与 `usePermission` Hook 配合，实现按钮级权限控制。

### 5.10 CI/CD — GitHub Actions 持续集成

**文件：** `.github/workflows/ci.yml`

**流程说明：**

每次 push 或 PR 到 main 分支时自动触发：

1. **Checkout** 拉取代码
2. **Setup Node 22** 安装指定 Node 版本
3. **npm ci** 干净安装依赖（锁定 lockfile，比 npm install 快且可复现）
4. **npm run lint** 代码风格检查（Oxlint）
5. **npm run typecheck** TypeScript 编译检查
6. **npm test** 跑全部 147 条单元测试 + E2E 测试
7. **npm run build** 生产构建

**为什么这么做：**
- 银行项目对代码质量要求极高，任何提交都必须经过类型检查、lint、测试三重关卡
- CI 跑完才能合并 PR，防止低质量代码进 main
- npm ci 锁定版本，避免"我本地能跑你不行"的环境差异

**如何一步步做的：**

1. 在仓库根目录创建 `.github/workflows/` 文件夹
2. 编写 `ci.yml`，定义 name / on / jobs 三要素
3. jobs 中用 `actions/checkout@v4` + `actions/setup-node@v4` 搭建基础环境
4. 先 `npm ci` 后再执行 `npm run lint` `npm run typecheck` `npm test` `npm run build`
5. push 到 GitHub 后自动触发，Actions 页面可实时查看流水线状态

**踩坑：**
- Windows runner 与 Linux runner 的路径分隔符差异：测试中部分 snapshot 路径用 `\\` 写死了，CI 跑在 ubuntu-latest 上路径变成 `/`。修复方案：测试中不用硬编码路径分隔符，改用 `path.join()` 或 `/` 统一使用 POSIX 风格。

### 5.11 Docker 容器化（生产模式）

**时间：** 第二版重构 2026-07-10

**背景：** 第一版 multi-stage 构建（node build → nginx 托管静态文件）完成后发现 MSW Mock 在纯静态文件模式下不工作，登录接口调不通。

**最终方案：** 放弃 MSW Mock，改为开发模式容器 + 生产模式容器双 Dockerfile 策略：

| 模式 | Dockerfile | 基础镜像 | 启动方式 | 特点 |
|------|-----------|---------|---------|------|
| 开发 | Dockerfile.dev | node:22-alpine | vite --host | MSW Mock 正常，与本地一致 |
| 生产 | Dockerfile | node:22-alpine → nginx:stable-alpine | nginx 托管 dist | 20MB 极简镜像，需真实后端 |

**生产模式 multi-stage Dockerfile 结构：**

```dockerfile
# Stage 1: build
FROM node:22-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: serve
FROM nginx:stable-alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**如何一步步做的：**

1. 编写 `Dockerfile`（multi-stage：build + nginx serve）
2. 编写 `.dockerignore` 排除 node_modules / .git / dist 避免无效层缓存
3. `docker build -t bank-admin .` 构建镜像（约 20MB 最终产物）
4. 测试时发现 MSW Mock 在纯静态 nginx 下被浏览器拦截，登录不可用
5. 改为开发模式：`docker run -p 5173:5173 bank-admin:dev` 直接跑 Vite dev server
6. 最终用 `docker-compose.yml` 编排单服务，简化启动命令

**踩坑：**
- **Docker Hub 网络不可达**：国内环境 pull node:22-alpine / nginx:stable-alpine 频繁超时，需要配置 Docker Desktop 镜像加速器（registry mirror）或提前 `docker pull` 缓存基础镜像
- **MSW Mock vs 真实后端**：生产模式 nginx 托管纯静态文件无法跑 MSW，需要配合真实后端 API。开发/演示场景用 dev 模式即可
- **Dockerfile 换行踩坑**：PowerShell 写入的 Dockerfile 被加了 Windows 换行（CRLF），Docker build 时部分指令解析异常。统一用 LF 换行符

**docker-compose.yml 生产模式极简版：**

```yaml
services:
  bank-admin:
    build: .
    ports:
      - "8080:80"
    restart: unless-stopped
```

### 5.12 E2E 测试 — Playwright

**时间：** 2026-07-10

**为什么用 Playwright 而非 Cypress：**
- Playwright 天然支持多浏览器（Chromium/Firefox/WebKit），测试覆盖面更广
- 代码风格接近 Vitest，学习成本低
- `webServer` 配置项可直接启动 Vite dev server，无需手动管理

**文件结构：**

```
e2e/
├── playwright.config.ts    # Playwright 配置（baseURL / webServer / workers）
├── login.spec.ts           # 登录流程（3 条用例）
└── users.spec.ts           # 用户管理（3 条用例）
```

**测试用例覆盖：**

**登录流程（3 条）：**
1. 使用正确凭据（admin/admin123）登录成功，跳转仪表盘
2. 错误凭据登录失败，显示错误提示
3. 未登录访问受保护页面（/users），重定向到登录页

**用户管理（3 条）：**
1. 用户列表加载正常，表格显示 10 条数据
2. 搜索关键字后表格正常过滤
3. 状态筛选下拉选择"正常"后表格正常过滤

**关键设计决策：**

```typescript
test.beforeEach(async ({ page }) => {
  // 从登录页开始完整登录
  await page.goto('/login');
  await page.getByPlaceholder('用户名').fill('admin');
  await page.getByPlaceholder('密码').fill('admin123');
  await page.getByRole('button', { name: '登 录' }).click();
  await expect(page).toHaveURL('/');
});
```

```typescript
// 使用内部导航而非 page.goto()
// 因为 Zustand 状态是 in-memory 的，goto() 会丢失验证状态
await page.getByText('用户管理').click();
await expect(page).toHaveURL('/users');
```

**如何一步步做的：**

1. `npx playwright init` 初始化 Playwright 配置
2. 编写 `e2e/playwright.config.ts`，配置 `baseURL: 'http://localhost:5173'` + `webServer` 自动启动 Vite
3. 先写 `login.spec.ts`：从登录页开始 → 填充表单 → 提交 → 验证跳转
4. 再写 `users.spec.ts`：登录后点击菜单进入用户管理 → 验证表格 → 搜索 → 筛选
5. 运行 `npx playwright test` 查看结果
6. 失败后查看 `test-results/` 下的截图定位问题
7. 修复后重跑直到全部通过

**踩坑记录：**

1. **Zustand 状态丢失（最核心的坑）**
   - 现象：`page.goto('/users')` 后页面停留在登录页，表格永不渲染
   - 原因：Zustand store 是 React 内存状态，`goto()` 导致页面完全重新加载，`authStore` 初始化为 `isAuthenticated: false`，`AuthGuard` 拦截重定向到 /login
   - 修复：登录后点击侧边栏菜单项（React Router 客户端导航）而非 `page.goto()`
   - 教训：理解 React 状态管理的作用域是 E2E 测试选择导航策略的前提

2. **Ant Design Select 组件的 Playwright 定位困难**
   - 现象：`getByPlaceholder('用户状态')` 找不到 Select 组件，`.ant-select-selector` 点击后无响应
   - 原因：SearchForm 中的 Select 在 Ant Design 6 默认使用虚拟列表渲染，Playwright 的下拉选项定位不准
   - 修复：改用 `getByRole('combobox')` 聚焦 + `page.keyboard` 键盘操作（ArrowDown + Enter）选择选项
   - 教训：Ant Design 复杂组件在 E2E 中优先用 role-based 定位 + 键盘操作，比 class 选择器稳定

3. **placeholder 文本不一致**
   - 现象：搜索框 placeholder 是 `搜索用户名/昵称/邮箱`，但测试用的是 `搜索用户名、昵称、邮箱`（中文顿号 vs 斜杠）
   - 修复：阅读实际代码确认 placeholder 文本后修正
   - 教训：测试中的定位文本必须与实际代码完全一致

**最终结果：**
```
Running 6 tests using 1 worker
  6 passed (15.4s)
```

## 六、踩坑汇总

1. **MSW v2 API 变更**：v2 使用 `http.get`/`http.post` 而非 v1 的 `rest.*`，官方文档有些不一致，需要多看 GitHub 示例。
2. **React 19 + Ant Design 6 兼容性**：两者都是最新大版本，初期遇到了一些类型不匹配问题，解决方案是锁定具体 minor 版本。
3. **TabsView setState during render**：React 严格模式下不允许 render 阶段调 setState，改用 useEffect 解决。
4. **ECharts Tree-shaking**：全量引入 ECharts 会增加 ~1MB 打包体积，改用按需引入（core + 具体 chart 类型 + 组件）后仅增加 ~80KB。
5. **Vitest 与 Vite 配置共享**：Vitest 扩展 Vite 配置，不需要像 Jest 那样单独维护 babel/jest 配置。
6. **身份证校验位算法**：加权因子 `[7,9,10,5,8,4,2,1,6,3,7,9,10,5,8,4,2]` + 模 11 映射 `["1","0","X","9","8","7","6","5","4","3","2"]`，需要仔细对照国标 GB 11643。
7. **PowerShell Out-File BOM 污染**：`Out-File -Encoding UTF8` 默认写 UTF-8 with BOM，JSON 解析器不认 `\uFEFF` 前缀，导致 `package.json` 被 Vite 的 PostCSS loader 拒读。后续统一改用 `[System.IO.File]::WriteAllText` 或 Node.js `fs.writeFileSync` 写入无 BOM 文件。
8. **echarts-for-react CJS/ESM 不兼容**：项目 `"type": "module"` 模式下 `echarts-for-react/lib/core`（CJS）被 Vite 错误解析，React 收到的组件是 `module.exports` 对象而非函数。改为自封装 `ChartContainer` 彻底规避第三方兼容性问题。
9. **ErrorBoundary 路由切换不重置**：React 错误边界捕获错误后 `hasError` 状态不会因 `<Outlet />` 切换而自动清除，导致切换页面仍显示旧错误页。通过 `getDerivedStateFromProps` 比较 `children.type` 检测路由切换并自动恢复。
10. **Node.js -e 多行字符串处理**：Windows 终端下 `node -e` 对多行字符串和转义字符处理极不稳定，后续统一改用 Node REPL MCP 或临时脚本文件执行复杂操作。
