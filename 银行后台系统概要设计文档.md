# 银行后台管理系统 — 概要设计文档

> **版本：** v1.0  
> **日期：** 2026-07-08  
> **技术栈：** React 19 + TypeScript 6 + Vite 8 + Ant Design 6 + ECharts 6 + Electron  
> **目标岗位：** 长亮科技 — 银行前端/全栈开发（薪资12k-20k）  
> **设计原则：** 生产级编码标准、可维护性、可扩展性、面试可解释性

---

## 一、架构总览

### 1.1 系统定位

银行后台管理系统（Bank Admin System）是银行内部运营人员使用的管理平台，处理用户管理、交易审核、账户操作、权限控制等核心金融业务。这套系统通常运行在内网环境中，但也需要通过 Electron 提供桌面端体验。

### 1.2 分层架构

`
┌─────────────────────────────────────────────────┐
│                   Electron Shell                │
│         窗口管理 / 托盘 / 自动更新 / 离线缓存        │
├─────────────────────────────────────────────────┤
│              Presentation Layer                  │
│   Pages / Components / Layouts / Hooks / Router  │
├─────────────────────────────────────────────────┤
│              Application Layer                    │
│   Zustand Store / React Query / Auth Context     │
├─────────────────────────────────────────────────┤
│                API Layer                          │
│   Axios Instance / Interceptors / Mock Service   │
├─────────────────────────────────────────────────┤
│                Data Layer                         │
│   MSW Mock / LocalStorage / IndexedDB (PWA)      │
└─────────────────────────────────────────────────┘
`

### 1.3 为什么这样分层

| 层 | 职责 | 面试可说的设计理由 |
|----|------|------------------|
| Electron Shell | 桌面端封装 | "银行内网系统需要 C/S 架构交付，Electron 允许复用 Web 代码" |
| Presentation | UI 渲染 + 交互 | "组件化 + 原子设计，保证 UI 一致性" |
| Application | 状态管理 + 业务编排 | "Zustand 比 Redux 轻量，适合中后台" |
| API | 网络通信 + 数据转换 | "统一拦截器处理 token 刷新和错误码" |
| Data | 数据持久化 | "MSW 让前后端并行开发，Mock 数据可平滑切换到真实后端" |

### 3.2 关键设计决策

`
┌────────────────────────────────────────────────────────┐
│  设计决策 1：为什么不选 Redux 而选 Zustand？              │
│                                                         │
│  面试话术："Zustand 的 API 更简洁，没有 Redux 的样板      │
│  代码（action type / reducer / dispatch），对于中后台     │
│  系统来说完全够用。而且 Zustand 的 TypeScript 支持更      │
│  自然，不需要额外的类型体操。"                            │
├────────────────────────────────────────────────────────┤
│  设计决策 2：用 MSW 做 Mock 而不是硬编码数据               │
│                                                         │
│  面试话术："MSW 在 Service Worker 层拦截请求，开发时      │
│  就像调真实后端一样。后期切换到真实接口时只需要改         │
│  baseURL，不需要改任何业务代码。"                        │
├────────────────────────────────────────────────────────┤
│  设计决策 3：为什么要有 components/ 和 pages/ 分离         │
│                                                         │
│  面试话术："通用组件放在 components/ 里跨页面复用，        │
│  pages/ 里只放页面特有的组合逻辑。这样当另一个系统         │
│  也需要表格组件时，直接引用就可以了。"                     │
├────────────────────────────────────────────────────────┤
│  设计决策 4：Electron 作为外壳而非核心                     │
│                                                         │
│  面试话术："银行要求桌面端交付，所以加 Electron 壳。       │
│  但核心业务代码是纯 Web，以后要改 B/S 架构只需移除        │
│  electron/ 目录，不改任何业务逻辑。"                      │
└────────────────────────────────────────────────────────┘
`

## 五、质量保障设计

### 5.1 代码质量

| 维度 | 措施 |
|------|------|
| 类型安全 | TypeScript strict 模式，禁止 any |
| 代码风格 | ESLint + Prettier，airbnb 规范 |
| 提交规范 | commitlint（feat/fix/chore/docs） |
| 组件规范 | 每个组件独立目录，含 index.tsx / types.ts / style.ts |

### 5.2 状态覆盖（面试重点）

每个页面组件必须覆盖以下状态：

`
┌─────────────────────────────────────────┐
│             页面状态矩阵                    │
├─────────────┬───────────────────────────┤
│  正常态      │ 数据正常加载和展示            │
│  空态        │ 无数据时的引导提示            │
│  加载态      │ Skeleton / Spin 占位        │
│  错误态      │ 网络错误 / 服务异常 + 重试     │
│  边界态      │ 超长文本截断 + Tooltip       │
│  无权限      │ 403 提示                    │
│  未登录      │ 重定向到登录页                │
└─────────────┴───────────────────────────┘
`

### 5.3 安全性设计

| 关注点 | 实现方式 |
|--------|---------|
| XSS 防护 | React 默认转义 + 不要 dangerouslySetInnerHTML |
| CSRF | Token 头 + SameSite Cookie |
| 敏感信息 | 金额脱敏（****1234）、手机号脱敏 |
| 操作审计 | 关键操作（审核/开户/冻结）写入操作日志 |
| 二次确认 | 删除/冻结/审核等操作弹窗确认 |
| 防重复提交 | 按钮 loading 状态 + 请求去重 |

## 八、面试速查表

### 被问到架构问题时

`
Q: "这个项目你用了什么状态管理？为什么？"
A: "Zustand。相比 Redux 样板代码少很多，中后台系统不需要 Redux 的中间件链和 immutable 更新，Zustand 足够用了。"

Q: "权限怎么做的？"
A: "RBAC 模型。用户 → 角色 → 权限点。菜单级用路由守卫控制，按钮级用自定义 Hook + 权限点判断。"

Q: "Mock 数据怎么弄的？"
A: "MSW（Mock Service Worker），在浏览器 Service Worker 层拦截请求，返回模拟数据。上线时只需要关掉 MSW，改成真实 API 地址就行。"

Q: "为什么用 Electron？"
A: "银行内网环境要求桌面端交付。Electron 可以复用 Web 代码，同时提供原生能力——托盘、通知、离线缓存。核心逻辑是纯 Web 的，以后如果要改 B/S 架构，把 electron/ 目录拿掉就行。"
`

### 被问到业务场景时

`
Q: "交易审核流程你们怎么设计的？"
A: "审核状态机：待审核 → 审核通过 / 审核驳回。通过后进入清算流程，驳回需要填写驳回原因。前端用 Tag 展示状态，审核操作有二次确认弹窗，防止误操作。"

Q: "表单验证怎么做的？"
A: "Ant Design Form 的 rules + 自定义 validator。金额字段限制两位小数、不允许负数。身份证号/手机号有正则校验。必填项有明确提示。"
`

### 被问到工程实践时

`
Q: "怎么保证代码质量？"
